# 生图性能检查与优化报告

日期：2026-07-13  
版本：0.6.8

## 结论

主要耗时分为三段：图片规划模型、插件本地前置检查、RunPod Worker 冷启动与生成。插件无法消除模型推理和 GPU 生成时间，但本轮已缩短本地检查，并优化多任务状态轮询；付费任务仍然按顺序提交到服务端队列，没有改成并行生图。

对实际 Endpoint 请求记录的复核表明，多图慢请求的主要耗时是 RunPod 队列等待，而不是浏览器端轮询：`Max workers = 1` 时，第二张会等待第一张完成。实际样本中，第一张约为 `15.8s delay + 27.02s execution`，同一 Worker 上第二张约为 `43.35s delay + 9.96s execution`。第二张执行时间明显下降，说明 Worker 内的热模型缓存已经生效；约 53 秒的总等待是第一张冷启动/加载和两张顺序推理的总和。

## 检查到的延迟来源

### 1. 目标稳定检查

旧流程每次尝试都会：

1. 读取聊天历史并构建完整图片上下文。
2. 固定等待。
3. 再次读取历史、状态和角色配置并重建完整上下文。

自动流程等待序列为 `140 + 620 + 320 + 320 = 1400ms`，手动流程为 `100 + 260 + 500 + 320 = 1180ms`。正常情况下通常第一轮即可确认，但发生其它插件延迟更新时，会重复较重的上下文构建。

### 2. 规划后的重复前台检查

图片规划模型返回后，旧流程会调用一次 `locate.findLastMessage()`，解析完成后、进入 RunPod 前又调用一次。两次检查之间没有需要保护的付费操作，因此第一笔查询是重复的。

### 3. 多任务状态轮询串行

`/run` 必须逐个提交以触发同一个 Endpoint 排队，这部分设计正确。旧流程随后也逐个等待每个 `/status` 网络响应；若有 N 个未完成任务，一轮状态检查约为 `N × 网络往返时间`。

### 4. 不应错误归因的耗时

- 图片规划模型的响应时间由所选 Connection Profile 和提示词长度决定。
- RunPod Worker 冷启动、模型加载与实际生成是总耗时的主要部分。
- Base64 图片写入 Extension Store 会产生本地序列化开销，但这是“每张图本地保存 90 天”的必要成本。
- 历史读取已有尾页指纹缓存；缓存命中时只读取 240 条尾页，不是每次都扫描完整聊天。

## 已实施优化

### 目标确认改用轻量宿主定位

现在每轮只构建一次完整上下文，等待后使用 TauriTavern 的 `locate.findLastMessage()` 在 Rust 侧确认最新助手消息及正文 hash。新的等待序列：

- 自动：`80 + 180 + 320 = 580ms`
- 手动：`40 + 120 + 240 = 400ms`

最坏固定等待分别减少 820ms 和 780ms，同时避免每轮第二次读取聊天运行状态和重建快照。

### 付费调用前只做一次最终检查

删除规划返回后的重复查询。现在解析停止标签、过滤无效位置、完成高费用确认后，只在 `/run` 前紧邻位置检查一次当前聊天、消息索引、正文 hash 和分支链。安全边界没有放松。

### 状态查询并发，生成提交保持排队

所有 `/run` 仍按提示词顺序连续 `await` 提交。只有无 GPU 生成作用的 `/status/{id}` 查询使用 `Promise.all` 同时读取。若有 N 个待完成任务，一轮轮询由约 `N × RTT` 降至约 `max(RTT)`。

### 保存 RunPod 服务端耗时

RunPod `/status` 返回的 `delayTime` 和 `executionTime` 现在会写入图片缓存记录，并在开启“测试模式通知”时连同客户端总等待一起显示。这样可以直接区分排队/冷启动、实际工作流执行和浏览器轮询时间，不再用总耗时猜测缓存是否生效。

### 暂停立即取消

魔法棒快捷按钮设为暂停时，会立即 Abort 当前图片管线并调用 RunPod `/cancel/{jobId}`。这不仅方便切换，也避免已经不需要的排队任务继续占用 GPU。

## 验证方式

- 单元测试断言所有 `/run` 调用都发生在第一个 `/status` 之前。
- 单元测试用同步门验证两个 `/status` 请求确实同时启动。
- 单元测试确认请求体不包含 Endpoint 未开放的 `negative_prompt`。
- `npm test`、`npm run check` 和 `git diff --check` 用于回归与语法检查。

未使用真实付费 Endpoint 做基准测试，因此报告中的毫秒数只包括代码中可确定的固定等待上限；网络、规划模型和 GPU 的真实耗时应在实际设备上通过“测试模式通知”记录。
